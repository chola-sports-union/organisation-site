# Registration Submission Pipeline Specification

This specification documents the backend integration for the Chola FC registration form submission, including serverless API handling, Zoho CRM storage, and Zoho workflow-driven email notifications.

---

## Purpose

The submission pipeline processes athlete registration applications by creating records in Zoho CRM via a serverless API and triggering Zoho workflows for email confirmations, staff notifications, and activity logging — all without the frontend communicating directly with Zoho.

---

## Requirements

### Requirement: Project Structure
The project **SHALL** use the following API directory structure:
```
api/
├── auth/
│   └── zoho/
│       ├── index.ts          # Initiates OAuth
│       └── callback.ts       # Handles OAuth callback
└── v1/
    └── registrations.ts      # Creates records in Zoho CRM
services/
└── zoho.ts                   # ZohoService class
```

### Requirement: OAuth 2.0 Setup (One-Time)
*   **GET** `/api/auth/zoho` **SHALL** redirect the browser to Zoho's authorization URL with `client_id`, `redirect_uri`, `response_type=code`, and appropriate `scope`.
*   **GET** `/api/auth/zoho/callback` **SHALL** receive the `code` parameter, exchange it for an access token and refresh token via `POST /oauth/v2/token`, and display the refresh token to the developer.
*   The authorized redirect URI **SHALL** be: `https://organisation-site-1r334f2ph-gowreesh10s-projects.vercel.app/api/auth/zoho/callback`
*   The refresh token **SHALL** be stored in Vercel environment variables and used by the Zoho service at runtime. The OAuth endpoints **SHALL NOT** be needed after initial setup.

### Requirement: Registration API Endpoint
*   A serverless function (Vercel Function) **SHALL** receive the POST request at `/api/v1/registrations`.
*   The request body **SHALL** contain all registration form fields as JSON.
*   The frontend **SHALL NOT** communicate directly with Zoho CRM.

### Requirement: Client-Side Validation
The form **SHALL** validate all required fields on submit before making any network request:
- First Name, Last Name, Date of Birth, Gender, Email, Phone, Address, Program, Football Experience **MUST** be non-empty.
- Email **MUST** match a valid email pattern.
- Phone **MUST** be exactly 10 digits with country code `+91`.
- Invalid fields **SHALL** display inline error messages.
- **IF** validation fails, **THEN** the form **SHALL NOT** proceed to the API call.

### Requirement: Loading State
*   **WHEN** the user submits a valid form,
*   **THEN** a loading spinner **SHALL** replace or overlay the submit button.
*   **AND** all form inputs **SHALL** be disabled during submission.
*   **AND** the submit button text **SHALL** change to a loading indicator (e.g., "Submitting...").
*   **AND** the user **SHALL NOT** be able to submit again while a submission is in progress.

### Requirement: Server-Side Validation & Business Rules
*   The API **SHALL** validate the incoming payload server-side.
*   Email **SHALL** be normalized: `trim()` and `lowercase()` before sending to Zoho.
*   Phone **SHALL** be validated as 10 digits with `+91` country code.
*   The API **SHALL** check the applicant's age from Date of Birth (for guardian enforcement only — DOB is stored, not age).
*   **IF** applicant age is below 18, **THEN** Parent/Guardian Name and Parent/Guardian Phone **MUST** be provided.
*   The API **SHALL NOT** generate or store a computed Age field — DOB is sufficient.
*   The API **SHALL NOT** store a custom Submission Date — Zoho's `Created Time` **SHALL** be used instead.
*   **IF** validation fails server-side, the API **SHALL** return a `400` status with error details.
*   **IF** successful, the API **SHALL** return a `201` status.

### Requirement: Duplicate Detection
*   **BEFORE** creating a record, the API **SHALL** search Zoho CRM by Phone **OR** Email.
*   **IF** a match is found, **THEN** the API **SHALL** return `409 Conflict` with an appropriate message (e.g., "An applicant with this phone or email is already registered.").
*   **IF** no match is found, **THEN** the API **SHALL** proceed with record creation.

### Requirement: Application ID
*   Application IDs **SHALL** be generated using Zoho CRM's Auto Number field feature.
*   The format **SHALL** be: `CFC-YYYY-NNNNNN` (e.g., `CFC-2026-000043`).
*   The backend **MAY** format the Auto Number for display if needed.
*   The Application ID **MUST NOT** be generated by counting records in Zoho (race condition risk).

### Requirement: Zoho CRM Record Creation
*   **WHEN** the API receives a valid payload with no duplicate detected,
*   **THEN** it **SHALL** create a record in Zoho CRM.
*   The record **SHALL** be created using a dedicated Zoho service layer (`services/zoho.ts`), not directly in the route handler.
*   The module **SHALL** be configurable via the `ZOHO_MODULE` environment variable.
*   The following fields **SHALL** be mapped:

| Field                | Zoho CRM Field       | Notes                              |
|----------------------|----------------------|------------------------------------|
| Application ID       | Application_ID       | Zoho Auto Number field             |
| First Name           | First_Name           |                                    |
| Last Name            | Last_Name            |                                    |
| DOB                  | Date_of_Birth        | Store DOB only, not age            |
| Gender               | Gender               |                                    |
| Email                | Email                | Normalized (trim, lowercase)       |
| Phone                | Phone                | 10 digits +91                      |
| Address              | Address              |                                    |
| Parent/Guardian Name | Guardian_Name        | Required for minors                |
| Parent/Guardian Phone| Guardian_Phone       | Required for minors                |
| Program              | Program              |                                    |
| Football Experience  | Football_Experience  |                                    |
| School Name          | School_Name          |                                    |
| Medical Conditions   | Medical_Conditions   |                                    |
| Motivation           | Motivation           | Statement of Purpose               |
| Status               | Status               | Default: "New" (see pipeline below)|
| Source               | Source               | Enum (see below)                   |

### Requirement: Source Enum
The Source field **SHALL** be an enum with the following values:
- Website
- Instagram
- Facebook
- Walk-in
- Referral
- Summer Camp
- School Visit

Default: `Website`.

### Requirement: Status Pipeline
The Status field **SHALL** follow this pipeline in Zoho CRM:
```
New
  ↓
Contact Attempted
  ↓
Contacted
  ↓
Trial Scheduled
  ↓
Trial Completed
  ↓
Admission Pending
  ↓
Joined
```
Terminal states: `Rejected`, `Closed`.

Default on creation: `New`.

### Requirement: Zoho Workflow Trigger
*   **WHEN** the Zoho CRM record has been created successfully,
*   **THEN** a Zoho workflow **SHALL** be triggered automatically.
*   The workflow **SHALL**:
    1. Send a confirmation email to the applicant.
    2. Notify academy staff at `cholafootballclub@gmail.com`.
    3. Create an activity/log entry in Zoho CRM.

### Requirement: Applicant Email Confirmation
*   The confirmation email **SHALL** be sent from the Zoho workflow using the organization's email.
*   The applicant email **SHALL** contain:
    *   **Subject:** "Registration Received – Chola FC"
    *   **Body:**
        ```
        Hello {First Name},

        Thank you for registering with Chola FC.

        We have received your application.

        Your Application ID is {Application ID}.

        Our team will contact you within 2 working days.

        Regards,
        Chola FC
        ```

### Requirement: API Response
*   **WHEN** the Zoho CRM record has been created successfully,
*   **THEN** the API **SHALL** return:
    *   Status: `201`
    *   Body:
        ```json
        {
            "success": true,
            "applicationId": "CFC-2026-000043",
            "crmRecordId": "123456789",
            "message": "Registration submitted successfully."
        }
        ```

### Requirement: Duplicate Conflict Response
*   **IF** a duplicate is detected (same phone or email),
*   **THEN** the API **SHALL** return:
    *   Status: `409`
    *   Body:
        ```json
        {
            "success": false,
            "error": "An applicant with this phone or email is already registered."
        }
        ```

### Requirement: Success Screen
*   **WHEN** the API returns a 201 response,
*   **THEN** the form **SHALL** be hidden and a success screen **SHALL** be displayed.
*   The success screen **SHALL** show:
    *   A checkmark or success icon.
    *   A heading: "Registration Submitted!"
    *   The Application ID.
    *   A message: "Thank you for registering with Chola FC. Our team will contact you within 2 working days."
    *   A "Submit Another Registration" button to reset the form.

### Requirement: Error Handling
*   **IF** the API call fails (network error or server error),
*   **THEN** the loading state **SHALL** be removed.
*   **AND** an error message **SHALL** be displayed to the user (e.g., "Something went wrong. Please try again.").
*   **AND** the form fields **SHALL** remain filled with the user's data so they can retry.

### Requirement: Non-Functional
*   Backend **MUST NOT** expose Zoho credentials.
*   All input **MUST** be validated server-side.
*   Duplicate submissions **MUST** be prevented via Zoho search before creation (409 on match).
*   Failures **SHALL** be logged.
*   The API **SHALL** remain independent of Zoho implementation details — Zoho logic **SHALL** live in `services/zoho.ts`, separate from the route handler — so another CRM can replace Zoho later without changing the frontend.
*   Environment variables **SHALL** be used for all secrets. `ZOHO_ACCOUNTS_URL` **SHALL** be `https://accounts.zoho.in` for the Indian region (or `https://accounts.zoho.com` for US).
*   REST API best practices **SHALL** be followed.

## Scenarios

### Scenario: Successful Registration
*   **GIVEN** a user is on the Join page with the registration form visible
*   **WHEN** they fill all required fields and click "Submit Registration"
*   **THEN** client-side validation passes
*   **AND** the submit button shows a loading spinner and becomes disabled
*   **AND** a POST request is sent to `/api/v1/registrations`
*   **AND** the server validates, normalizes email (trim, lowercase), and validates phone (10 digits, +91)
*   **AND** the server searches Zoho by phone and email — no duplicate found
*   **AND** the server creates a record in Zoho CRM via the service layer
*   **AND** Zoho Auto Number generates the Application ID
*   **AND** Zoho workflow triggers (applicant email, staff notification, activity log)
*   **AND** the server returns 201 with `applicationId`, `crmRecordId`, and success message
*   **THEN** the form is hidden and the success screen is shown with the Application ID

### Scenario: Duplicate Registration Detected
*   **GIVEN** a user submits the registration form
*   **WHEN** their phone or email matches an existing Zoho record
*   **THEN** the API returns 409 Conflict
*   **AND** the error message is displayed: "An applicant with this phone or email is already registered."

### Scenario: Minor Applicant Requires Guardian Info
*   **GIVEN** a user is on the Join page
*   **WHEN** they enter a Date of Birth making them under 18
*   **AND** they submit without filling Parent/Guardian fields
*   **THEN** server-side validation returns a 400 error
*   **AND** the error message indicates guardian fields are required for minors

### Scenario: Network Failure During Submission
*   **GIVEN** a user submits the registration form
*   **WHEN** the POST request fails due to a network error
*   **THEN** the loading spinner is removed
*   **AND** an error message is displayed: "Something went wrong. Please try again."
*   **AND** the form fields remain filled
*   **AND** the user can retry submission

### Scenario: Server-Side Validation Failure
*   **GIVEN** a user submits the registration form
*   **WHEN** the server returns a 400 status with validation errors
*   **THEN** the loading spinner is removed
*   **AND** the specific error message is displayed to the user
*   **AND** the form fields remain filled for correction

### Scenario: Resetting After Success
*   **GIVEN** a user is viewing the success screen
*   **WHEN** they click "Submit Another Registration"
*   **THEN** the success screen is hidden
*   **AND** all form fields are reset to their default values
*   **AND** the registration form is shown again
